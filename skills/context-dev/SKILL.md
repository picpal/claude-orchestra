# Development Context

개발 작업에 집중하는 컨텍스트입니다.

## 핵심 원칙

### 1. "Write code first, explain after"
- 설명보다 동작하는 코드 우선
- 코드로 보여준 후 필요시 설명

### 2. "Prefer working solutions over perfect solutions"
- 완벽함보다 동작하는 솔루션
- 점진적 개선 (리팩토링)

### 3. 코드 변경 후 테스트 실행
- 모든 변경 후 관련 테스트 실행
- 테스트 실패 시 즉시 수정

### 4. 원자적 커밋 구조 유지
- TODO 단위로 커밋
- 논리적으로 분리된 변경

## 우선순위

```
1위: 기능성 (Functionality)
     "일단 동작해야 한다"

2위: 정확성 (Correctness)
     "올바르게 동작해야 한다"

3위: 코드 품질 (Code Quality)
     "잘 작성되어야 한다"
```

## 권장 도구

### 코드 작성
- **Edit**: 기존 파일 수정
- **Write**: 새 파일 생성

### 실행/검증
- **Bash**: 테스트, 빌드, Git 명령

### 탐색
- **Grep**: 코드 패턴 검색
- **Glob**: 파일 찾기
- **Read**: 파일 내용 확인

## 워크플로우

```
1. 요구사항 확인
    ↓
2. 관련 코드 탐색 (Grep/Glob)
    ↓
3. 테스트 작성 [TEST]
    ↓
4. 테스트 실행 (실패 확인)
    ↓
5. 구현 [IMPL]
    ↓
6. 테스트 실행 (성공 확인)
    ↓
7. 리팩토링 [REFACTOR]
    ↓
8. 검증 (/verify)
    ↓
9. 커밋
```

## 금지 사항

### DO NOT
- ❌ 테스트 없이 구현
- ❌ 범위 외 "개선"
- ❌ 불필요한 추상화
- ❌ 미완성 코드 커밋
- ❌ 테스트 삭제/스킵

### AVOID
- ⚠️ 과도한 주석
- ⚠️ 복잡한 한 줄 코드
- ⚠️ 깊은 중첩
- ⚠️ any 타입

## 체크리스트

### 코드 작성 전
- [ ] 요구사항 명확히 이해
- [ ] 관련 기존 코드 확인
- [ ] 테스트 케이스 계획

### 코드 작성 중
- [ ] TDD 사이클 준수
- [ ] 명확한 명명
- [ ] 에러 핸들링

### 코드 작성 후
- [ ] 모든 테스트 통과
- [ ] 린트 통과
- [ ] 타입 체크 통과
- [ ] 커버리지 확인

## 유용한 명령어

```bash
# 테스트 실행
npm test

# 특정 파일 테스트
npm test -- auth.test.ts

# 커버리지
npm test -- --coverage

# 타입 체크
npx tsc --noEmit

# 린트
npx eslint .
```
